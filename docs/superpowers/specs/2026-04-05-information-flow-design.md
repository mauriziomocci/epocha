# Information Flow Module -- Design Spec

## Goal

Implement the information flow system so that agents no longer live in informational isolation. Actions, events, and rumors propagate through the social network with realistic delay, distortion, and filtering. An agent in a remote corner of the society learns about a betrayal days after it happened, and the version they hear may be exaggerated or softened depending on who told them.

## Architecture

Three independent modules with clear interfaces, plus minimal integration into the existing tick engine.

### Propagation Engine (`epocha/apps/agents/information_flow.py`)

Runs once per tick, after agent decisions and before memory decay. Collects propagable information (significant actions and public events) and spreads it 1 hop through the social network per tick.

**What propagates:**

- **Agent actions** with `emotional_weight >= 0.3` (from `_ACTION_EMOTIONAL_WEIGHT` in engine.py). In practice: argue (0.4), help (0.3), betray (0.8). Routine actions like work (0.1), rest (0.05), socialize (0.2) do not propagate -- nobody gossips about someone resting.
- **Public events** (injected by the user or generated by the system via Event model) propagate instantly to all living agents with source_type "public", reliability 1.0, no distortion.

**Channels:**

| Channel | Trigger | Recipients | Hops | Distortion | Reliability |
|---------|---------|------------|------|------------|-------------|
| direct | Agent performs the action | The agent itself | 0 | None | 1.0 |
| hearsay | Significant action by a socially connected agent | Agents with direct Relationship to the actor | 1 per tick | Yes | 0.7 |
| rumor | Hearsay propagates beyond first hop | Agents connected to the transmitter | 2+ per tick | Yes, cumulative | 0.7^hop |
| public | System event (injected or generated) | All living agents | 0 (instant) | None | 1.0 |

The difference between hearsay and rumor is purely the hop count: hop 1 is hearsay, hop 2+ is rumor.

**Algorithm per tick:**

```
1. Collect "direct" memories created THIS tick with emotional_weight >= 0.3
2. For each propagable memory:
   a. Find agents connected to the protagonist via Relationship (is_alive=True)
   b. For each connected agent:
      - Skip if agent already has a hearsay/rumor memory for the same
        origin_agent + action in this tick (deduplication)
      - Apply distortion (distortion engine, using protagonist's personality)
      - Apply belief filter (receiver's personality, reliability, relationship)
      - If accepted: create Memory with source_type="hearsay", degraded reliability
3. Collect "hearsay" memories created in the PREVIOUS tick (tick - 1)
   - Same process, but source_type becomes "rumor", reliability degrades further
   - Distortion uses the TRANSMITTER's personality (the agent who holds the hearsay)
4. No further propagation in the same tick (prevents combinatorial explosion)
```

**Reliability degradation per hop:**

```
hop 0 (direct):  reliability = 1.0
hop 1 (hearsay): reliability = 0.7
hop 2 (rumor):   reliability = 0.7 * 0.7 = 0.49
hop 3 (rumor):   reliability = 0.49 * 0.7 = 0.34
```

The 0.7 factor per hop is derived from serial reproduction research (Bartlett, 1932. "Remembering: A Study in Experimental and Social Psychology." Cambridge University Press), which demonstrates approximately 30% detail loss at each transmission step in communication chains.

**Maximum propagation depth:** 3 hops. At hop 3, reliability (0.34) falls below the belief filter threshold for most agents, so propagation self-extinguishes naturally.

**Propagation cap:** maximum 20 recipients per memory per tick (`_MAX_PROPAGATION_RECIPIENTS = 20`). Prevents unbounded fan-out in highly connected networks.

### Distortion Engine (`epocha/apps/agents/distortion.py`)

Pure function: takes content string and transmitter personality dict, returns distorted content string. Rule-based, deterministic, zero LLM calls.

**Interface:**

```python
def distort_information(content: str, personality: dict) -> str
```

**Rules based on Big Five traits:**

| Trait | High (>0.7) | Low (<0.3) |
|-------|-------------|------------|
| Neuroticism | Amplifies negativity: "argued" -> "fought bitterly" | Minimizes: "fought" -> "had a disagreement" |
| Agreeableness | Softens: "betrayed" -> "disappointed" | Exaggerates conflict: "argued" -> "attacked" |
| Openness | Embellishes, adds speculation ("perhaps because...") | Simplifies, removes nuance |
| Extraversion | Exaggerates scale: "someone complained" -> "everyone is complaining" | Understates: "the crowd protested" -> "a few people protested" |
| Conscientiousness | Preserves details, adds precision | Omits details, confuses facts |

**Calibration rules:**

- **Activation threshold**: a trait only distorts when extreme (>0.7 or <0.3). Between 0.3 and 0.7, the trait is neutral -- the agent transmits faithfully on that dimension.
- **Distortion strength**: proportional to the distance from the threshold. Neuroticism 0.95 distorts much more than 0.75.
- **Maximum 2 active traits**: if an agent has 3+ extreme traits, only the 2 furthest from the neutral zone apply. Prevents illegible compound distortions.
- **Pattern substitution**: not hardcoded strings, but parametric templates. Example: `"argued"` maps to `["had a disagreement", "fought bitterly", "attacked viciously"]` indexed by distortion strength (low/medium/high).

**Scientific basis:** The influence of personality traits on information transmission is supported by research on rumor psychology (Allport & Postman, 1947. "The Psychology of Rumor." Henry Holt and Company) which identifies three processes in serial transmission: leveling (loss of detail), sharpening (emphasis of select details), and assimilation (distortion toward the transmitter's expectations and personality).

### Belief Filter (`epocha/apps/agents/belief.py`)

Pure function: decides whether an agent accepts or discards received information. Returns a boolean.

**Interface:**

```python
def should_believe(
    reliability: float,
    receiver_personality: dict,
    relationship_strength: float,
    relationship_sentiment: float,
) -> bool
```

**Formula:**

```
relationship_trust = (relationship_strength + max(0, relationship_sentiment)) / 2
personality_factor = agreeableness * 0.6 + openness * 0.4
acceptance_score = (reliability * 0.4) + (relationship_trust * 0.3) + (personality_factor * 0.3)
```

When there is no relationship between transmitter and receiver, `relationship_trust` defaults to 0.3 (stranger: neither trust nor distrust).

**Acceptance threshold:** 0.4. Below this value, the agent discards the information and no memory is created.

**Behavioral examples:**

| Scenario | reliability | rel_trust | personality | score | Accepts? |
|----------|-----------|-----------|-------------|-------|----------|
| Trusted friend relays fresh hearsay | 0.7 | 0.8 | 0.5 | 0.67 | Yes |
| Stranger passes third-hand rumor | 0.3 | 0.3 | 0.5 | 0.36 | No |
| Weak rumor but very gullible agent | 0.3 | 0.3 | 0.9 | 0.48 | Yes |
| Reliable news but very skeptical agent | 0.7 | 0.5 | 0.2 | 0.49 | Yes (barely) |

**Scientific basis:** The weight of interpersonal relationships on information credibility is supported by the integrative model of trust (Mayer, Davis & Schoorman, 1995. "An Integrative Model of Organizational Trust." Academy of Management Review, 20(3), 709-734). The influence of agreeableness on credulity is documented in Graziano & Tobin (2002. "Agreeableness: Dimension of Personality or Social Desirability Artifact?" Journal of Personality, 70(5), 695-728).

## Model Changes

**Memory model** -- one new field:

```python
origin_agent = models.ForeignKey(
    "Agent", null=True, blank=True, on_delete=models.SET_NULL,
    related_name="originated_memories",
    help_text="Agent who originally performed the action (for dedup and traceability)",
)
```

- For "direct" memories: `origin_agent = agent` (the agent themselves)
- For "hearsay"/"rumor": `origin_agent` = the agent who performed the original action
- For "public": `origin_agent = None` (system event, no single origin)

Used for deduplication: before creating a hearsay/rumor memory, check if the receiver already has one with the same `origin_agent` + similar action in the current tick.

Requires one migration.

## Settings

```python
# Minimum emotional weight for an action to propagate through the social network.
# Actions below this threshold are considered too mundane for gossip.
# Source: threshold aligns with _ACTION_EMOTIONAL_WEIGHT in engine.py where
# socialize=0.2 (excluded) and help=0.3 (included).
EPOCHA_INFO_FLOW_PROPAGATION_THRESHOLD = 0.3

# Reliability decay factor per hop in the communication chain.
# Source: Bartlett (1932) serial reproduction experiments show ~30% detail loss per step.
EPOCHA_INFO_FLOW_RELIABILITY_DECAY = 0.7

# Maximum hops before information stops propagating.
EPOCHA_INFO_FLOW_MAX_HOPS = 3

# Belief filter acceptance threshold (0.0-1.0).
EPOCHA_INFO_FLOW_BELIEF_THRESHOLD = 0.4
```

All settings go in `config/settings/base.py` with sensible defaults. Overridable per environment.

## Integration into Tick Engine

One line added to `SimulationEngine.run_tick()` and to the Celery tick loop, between agent action processing and memory decay:

```python
from epocha.apps.agents.information_flow import propagate_information
propagate_information(simulation, tick)
```

No other changes to existing code. Hearsay and rumor memories are standard Memory objects -- they are automatically picked up by `get_relevant_memories`, displayed in the activity feed, and included in agent decision context. The `source_type` label (already shown in parentheses in the context) tells the agent and the user whether the information is direct, hearsay, or rumor.

## Computational Cost

With 20 agents and ~3 significant actions per tick:
- Hop 1: ~3 actions * ~5 connected agents = ~15 belief filter evaluations
- Hop 2: ~8 accepted hearsays * ~4 connected agents = ~32 evaluations
- Total: ~50 filter evaluations + ~20 memory creates per tick
- Zero LLM calls
- Negligible compared to the 20 LLM calls for agent decisions

## Future Extension

The distortion engine interface (`distort_information(content, personality) -> str`) is designed to be swappable. A future LLM-based implementation could replace the rule-based engine via a setting, using the chat provider with batched calls per transmitter. This was evaluated and deferred for cost reasons -- see memory note `project_info_flow_llm_distortion.md`.
